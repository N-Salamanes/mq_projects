COMP329 Assignment 2: iKnow - A Tutoring System for COMP329

COMP329 Assignment 2

1. Introduction

In this assignment you will implement iKnow329, a prototype of an intelligent tutoring system, in SWI Prolog (version 7.2.3). iKnow329 should ask a student questions about three predefined topics taught in COMP329 (Knowledge Systems), evaluate the student's answer to a question, give hints if a student has difficulty to answer a certain type of questions, inform the student about the outcome of a run during a session, and store the results at the end of a session for subsequent inspection by a teacher. That means the system should provide two modes: a student mode where the student interacts with the tutoring system and a teacher mode where the teacher can review the results of a student.

First you will implement a basic version of iKnow329 that you can then take as a starting point for the full version that provides broader coverage and additional features.

As you can see, after the start (?- start.) the student has to provide the student ID for a session. For each run during a session, the tutoring system asks the student to select a level of difficulty (easy, medium, or hard) and to choose a topic (in our case: parsing or probability theory or Prolog). The tutoring system then asks the student exactly 5 questions. Some of these questions have the form of yes/no-questions and some have the form of wh-questions. The answer to a yes/no-question is always yes or no (or one of its variations). The answer to a wh-question (what, why, where, how, ...) can be one or more words or even an entire sentence. As you would expect, wh-questions are more challenging to process since you have to do some advanced pattern matching. A run ends with a laudatio where the system informs the student about the result of the run and asks if the student wants to continue or not.

Of course, things do not go always so smoothly and a student might fail to answer a question (see question 2 and 4 below). In the case of a wh-question, the tutoring system should offer a hint, if the student provides the wrong answer (as illustrated below for question 4). However, if the student asks for a hint and subsequently answers the question correctly, then the student will not get full marks for that question (in our case half a mark will be deduced for an easy question). On the other hand, if the answer to a wh-question was wrong and the student does not accept a hint, then you give 0 marks for that question and move on to the next question (if there is still one left).

In order to implement the basic version of this tutoring system, you only have to implement the difficulty level 'easy' and the topic 'Prolog' (but your system has to display a warning if the student selects a different difficulty level and/or topic that has not been implemented for the basic version of the system).

You have to use the file (kterms.pl) for the basic version of this assignment that comes with the starter kit (assignment-2-starter-kit.zip). This start starter kit consists of three files: iKnow329.pl, kterms.pl, and users.pl. The files kterms.pl and users.pl are loaded automatically at the start of the system (iKnow329.pl). The initial file kterms.pl consists of exactly those five knowledge terms (kterm/6) that have been used in the "easy Prolog" session above. These knowledge terms have 6 arguments: number, topic, level, question, answer, and hint; each one represented as a term. The term question/3 has three arguments: question type, actual question, mark for the question. The term answer/2 has two arguments: the correct answer and a list of acceptable answers that can be tested against the input of the student. The term hint/2 has two arguments: one for the hint to be displayed and one for the value to be deduced from the student's marks if the hint has been used. As already explained, it only makes sense to use a hint in the context of a wh-question.

% -----------------------------------------------------------
% kterm/6
% -----------------------------------------------------------

kterm(number(1), 
      topic(prolog), 
      level(easy), 
      question(wh, 'What does Prolog stand for?', 1), 
      answer('Programming in Logic', [[programming, in, logic], [logic, programming]]), 
      hint('Pro... in Log...', 0.5)).

kterm(number(2), 
      topic(prolog), 
      level(easy), 
      question(yn, 'Is Prolog a procedural programming language?', 1), 
      answer('no', [[no], [nope], [nah]]), 
      hint(nil, nil)).

kterm(number(3), 
      topic(prolog), 
      level(easy), 
      question(yn, 'Does the following unification succeed: [a, [b]|[c, d]]=[X, Y, Z]?', 1), 
      answer('yes', [[yes], [yep], [sure]]),  
      hint(nil, nil)).

kterm(number(4), 
      topic(prolog), 
      level(easy), 
      question(wh, 'What is the most common data structure in Prolog?', 1), 
      answer('lists', [[list], [lists]]), 
      hint('It has a head and a tail.', 0.5)).

kterm(number(5), 
      topic(prolog), 
      level(easy), 
      question(wh, 'Why is Prolog good for AI programming?', 1), 
      answer('Because Prolog is good at pattern matching', [[pattern, matching], ['pattern-matching']]), 
      hint('Regular expressions provide a similar functionality.', 0.5)).

The file (users.pl) that is also automatically loaded at the start of the system needs to be maintained during a session. This file contains only three student IDs (s1, s2, and s3) and one teacher ID (t1). We don't worry about more detailed user information and passwords for our prototype. However, we worry about the students' history and achievements during the tutoring sessions. At the beginning, the file users.pl adds the following information to the knowledge base: 

student(s1,[]).
student(s2,[]).
student(s3,[]).

teacher(t1).
consisting of two types of facts: student/2 and teacher/1. The information in users.pl is loaded at the start of a session into the knowledge base, updated in the knowledge base after each run and then written to the file users.pl at the end of a session. The second argument of student/2 is used to store a student's history and achievements. In our case, the file users.pl looks as follows at the end of our session: 

student(s1,[[prolog, easy, 3.5], [prolog, easy, 5]]).
student(s2,[]).
student(s3,[]).

teacher(t1).

Note that the user ID starts either with an "s" for student or a "t" for teacher, the tutoring system should branch either into the student mode or the teacher mode according to the provided input, and prompt the user again if the user provides a wrong input.

The top predicate start/0 of iKnow329.pl shows the two files (kterms.pl and users.pl) that are loaded for each session, the two modes that the system has to support and the file (users.pl) that needs to be updated after each session.

% -----------------------------------------------------------
% start/0
% -----------------------------------------------------------

start :-
  consult('kterms.pl'),
  consult('users.pl'),
  welcome,
  login(ID, Mode),
  (
     Mode = student
     ->
     main_process_student(ID)
  ;
     Mode = teacher
     ->
     main_process_teacher(ID)
  ),
  export_user_knowledge_base('users.pl').

The implementation of the basic version of iKnow329 is worth 12 marks. Please check the marking schema below for the distribution of these marks.

3. Full Version of iKnow329

In order to implement the full version of the tutoring system, you have to provide additional data for three topics and three difficulty levels. You already have data for one topic (Prolog) and one difficulty level (easy) as 5 knowledge terms are already in kterms.pl. In order to complete the implementation, you have to add 40 additional knowledge terms that encode different pieces of knowledge. Prolog is one topic that the tutoring system has to support, the other two topics are parsing and probability theory. At least 20 of these 40 knowledge terms should contain wh-questions. You have to use the following marking schema to process the hints of wh-questions for these different difficulty levels (remember the cost for a hint of an easy question is 0.5 mark):

  • 1 mark for an easy question answered correctly but only 0.5 mark if the students used the hint;
  • 2 marks for a medium question answered correctly but only 1 mark if the student used the hint;
  • 3 marks for a hard question answered correctly but only 1.5 mark if the sutdent used the hint.

This extension is worth 5 marks.

You can get an additional 3 marks for this assignment for the implementation of a specific feature that you think is useful and adds value to the tutoring system. For example, a mechanism that tries to correct the user input automatically in case of misspelling might be very helpful or more detailed information about a student's history and achievements (the current information is indeed very rudimentary). If you choose to add such an additional feature to the tutoring system, you will need to document this feature in a two-page report and explain why this feature adds value to your system.
